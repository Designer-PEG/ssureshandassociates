## Core Design Principles

### 1. User-Centered Design

Every design decision must trace back to a real user need. This is not aspirational — it is operational.

- Begin with research: jobs-to-be-done, pain points, mental models, and existing behavior patterns
- Map user journeys before laying out pages; understand entry points and exit intentions
- Validate assumptions through usability testing, heatmaps, and session recordings
- Prioritize task completion over aesthetic novelty
- Design for the 80% use case while accommodating edge cases gracefully

**Checklist:**
- [ ] User personas or archetypes defined and referenced
- [ ] Primary user tasks identified per page
- [ ] Content hierarchy reflects user priorities, not internal org structure
- [ ] Error recovery paths designed with empathy

---

### 2. Visual Hierarchy

Hierarchy directs the eye and communicates importance without the user consciously noticing.

- Use **size, weight, contrast, color, and spacing** as hierarchy tools — never use them arbitrarily
- The most important element on any page should be visually dominant; the second most important should be clearly subordinate
- Limit heading levels to what is semantically and visually necessary
- Headlines should be scannable at 2–3× normal reading distance

**Rules:**
- One H1 per page, always
- Avoid more than 3 levels of typographic hierarchy on a single viewport
- Never use visual emphasis (bold, color, size) on more than 10–15% of text content
- If everything is emphasized, nothing is

---

### 3. Accessibility (WCAG 2.2)

Accessibility is not a checklist item added at the end — it is a design constraint that improves experiences for all users.

- Design with contrast, focus states, and keyboard navigation from the first wireframe
- Screen reader compatibility requires semantic HTML; accessibility is built in code, not just visuals
- Every interactive element must have a visible focus indicator
- Motion and animation should respect `prefers-reduced-motion`

*Full details in the [Accessibility Standards](#accessibility-standards) section.*

---

### 4. Mobile-First Design

Start design at the smallest viewport and progressively enhance for larger screens.

- Mobile is often the primary or first touchpoint; do not treat it as a stripped-down afterthought
- Touch targets must be at least 44×44px (Apple HIG) or 48×48px (Material Design)
- Thumb zone ergonomics: primary actions belong in the lower two-thirds of the screen
- Avoid hover-only interactions; ensure all functionality is accessible via tap
- Content priority on mobile must be ruthless — fewer elements, larger text, more whitespace

---

### 5. Responsive Layouts

Every layout must adapt gracefully across viewport sizes without breaking information hierarchy.

- Use CSS Grid and Flexbox as the foundation; avoid fixed-width containers where possible
- Components should have defined behavior at each breakpoint — not just "shrink to fit"
- Images must be responsive with defined `srcset` or modern CSS `clamp()` for sizing
- Text line lengths should stay between 55–75 characters (the typographic sweet spot) at all sizes

*Full breakpoint specifications in the [Responsive Design Standards](#responsive-design-standards) section.*

---

### 6. Design Systems

Design systems reduce inconsistency, speed up production, and enforce quality at scale.

- Define tokens (colors, spacing, type scale, border radius, shadows) before designing components
- Components should be self-contained, documented, and cover all states (default, hover, focus, active, disabled, error)
- Never design one-off components if a system component can be adapted
- Systems must be living documents — outdated documentation is worse than none

*Full specifications in the [Design System Standards](#design-system-standards) section.*

---

### 7. Component Consistency

Consistent patterns reduce cognitive load and build user trust.

- The same action should always look and behave the same way across the product
- Destructive actions (delete, cancel) must be visually distinct from constructive ones
- Use established patterns (hamburger menus, underlined links, blue for links) only where convention is a feature
- When breaking convention, do so intentionally and clearly

---

### 8. Cognitive Load Reduction

Every element on screen competes for mental attention. Design as if attention is scarce — because it is.

- Remove anything that does not serve a clear purpose on that page for that user at that moment
- Use progressive disclosure: show only what is needed to complete the next step
- Chunk related information into visually grouped units (Gestalt: proximity and similarity)
- Forms should ask only what is absolutely necessary; every extra field reduces conversion
- Eliminate redundant labels, decorative copy, and filler imagery

---

### 9. Conversion-Focused Design

Design exists within a business context. Helping users accomplish goals and helping businesses achieve outcomes are not in conflict — they are aligned.

- Every page should have one primary conversion goal and be designed around it
- CTAs must be prominent, specific, and benefit-oriented ("Start my free trial" > "Submit")
- Reduce friction on high-intent paths: minimize clicks, pre-fill where possible, remove distractions
- Use social proof, urgency, and specificity to close hesitation gaps
- A/B test key conversion elements systematically; do not rely on intuition alone

---

### 10. UX Writing Principles

Words are interface elements. Copy is design.

- Write in the second person ("You'll get…") to create personal relevance
- Use active voice; passive voice signals institutional distancing
- Lead with benefit, follow with feature: "Save 10 hours a week — automate your reports"
- Microcopy (button labels, error messages, empty states) deserves as much attention as headlines
- Error messages must explain what went wrong, why, and how to fix it — never just "Invalid input"
- Avoid jargon unless it is the user's language; use plain language by default

---

### 11. Trust-Building Techniques

Trust is earned through consistency, transparency, and credibility signals — not decoration.

- Social proof is most effective when specific: real names, photos, company, and results
- Certifications and security badges near sensitive actions (checkout, form submission) reduce anxiety
- Transparency about pricing, process, and limitations reduces fear and builds confidence
- Avoid dark patterns; manipulative design destroys trust permanently when noticed

---

### 12. Performance-Aware Design

Visual design decisions have real performance implications. Design with this awareness.

- Prefer CSS effects (gradients, shadows, blur) over heavy image assets wherever possible
- Web fonts add latency; limit to 2 families with a maximum of 3–4 weights total
- Hero images must be properly sized, formatted (WebP/AVIF), and lazy-loaded below the fold
- Animations driven by `transform` and `opacity` are GPU-accelerated; avoid animating layout properties
- Every decorative element should be weighed against its performance cost

## Homepage Design Guidelines

### Purpose

The homepage is the highest-traffic, highest-stakes page in most web products. It must:

1. **Create a strong first impression** within 3–5 seconds — the window before a user decides to stay or leave
2. **Communicate the core value proposition** clearly and immediately without requiring scroll
3. **Orient the user** — tell them what this product/service does, who it's for, and why it matters
4. **Drive conversion** — move the right users toward the next meaningful action
5. **Filter audience** — homepage should attract the right users and politely repel mismatched ones

> *A homepage is not a brochure. It is a guided experience designed to convert interest into intent.*

---

### Hero Section

The hero is the most critical real estate on the web. Every word and pixel must earn its place.

#### Headline

- Maximum 8–10 words for primary headline; longer if the product complexity demands it
- Structure: **[Outcome] for [Audience]** or **[Verb] your [Problem]**
- Avoid puns, abstract metaphors, or cleverness that obscures meaning
- The headline should work without the subheading — test it in isolation
- Examples of strong patterns:
  - "Ship faster. Break less." (benefit + reassurance)
  - "Analytics for teams that move fast" (outcome + audience)
  - "The design tool for product teams" (category + audience)

#### Value Proposition

- The subheading (1–2 sentences) should expand on the headline with concrete specificity
- Answer: What is it? Who is it for? What does it do that alternatives don't?
- Avoid "powerful," "seamless," "next-generation," and other content-free adjectives
- Use numbers when possible: "Used by 40,000 designers" beats "Used by thousands"

#### Supporting Copy

- Optional third tier of copy for nuance or context
- If included, keep to 1–2 sentences; do not repeat the value proposition
- This is the place to pre-empt objections or address the "catch" a skeptical user anticipates

#### CTA Placement and Hierarchy

- **Primary CTA**: Placed in the hero, above the fold, high contrast, specific label
- **Secondary CTA**: Lower visual weight; alternative path (e.g., "Watch demo" vs. "Start free trial")
- Never more than 2 CTAs in the hero; competing calls create decision paralysis
- CTA label should describe the outcome, not the action: "Get my free audit" > "Submit"
- Reinforce with micro-copy beneath CTA: "No credit card required" / "Cancel anytime" / "Free for 14 days"

#### Hero Imagery

- Product screenshots convert better than abstract illustrations for SaaS products
- Human faces increase emotional connection and dwell time — use real people, not stock-photo smiles
- Hero image must complement, not compete with, the headline
- Ensure sufficient contrast between the image and overlaid text at all times
- Dark mode consideration: ensure hero imagery reads in both modes or provide separate assets

#### Social Proof Placement in Hero

- Logo bar immediately below hero ("Trusted by teams at [Logo] [Logo] [Logo]") reduces bounce rates significantly
- A single high-credibility quote with attribution can appear below the CTA
- Star ratings or review counts (e.g., "4.9/5 from 2,400 reviews") add immediate credibility

---

### Navigation

#### Logo Positioning

- Top-left is the universal convention; deviation requires strong justification
- Logo should link to the homepage from all interior pages
- Logo area should be no more than 200px wide at desktop; reduce appropriately on mobile

#### Menu Structure

- Maximum 5–7 primary navigation items; more creates cognitive overload
- Use mega menus only when sub-navigation genuinely requires them; avoid for complexity theater
- Navigation labels must be nouns or noun phrases, not verbs (except CTA items)
- Order: most-visited or most-important pages left-to-right; CTA far right
- Active state must be visually distinct; users need to know where they are

#### Sticky Navigation

- Sticky nav is appropriate when pages are long (homepage, landing pages, documentation)
- Sticky nav must be visually differentiated from the scroll state (subtle shadow or background change)
- On scroll, reduce nav height by 10–15% to maximize content real estate
- Sticky nav must not obscure anchor links or fixed elements on mobile

#### Search Considerations

- Add search when the site has more than ~20 pages of distinct content
- Search icon alone is acceptable for secondary navigation; prominent search bar for content-heavy sites
- Search must return accurate results with graceful empty states

#### Mobile Navigation

- Hamburger menu is acceptable; label it "Menu" for accessibility and clarity
- Navigation drawer should open from top or right; never from left (conflicts with back gesture)
- Close button must be visible without scrolling; or close on outside tap
- Mobile nav should reduce to a maximum of 5 items; consider bottom nav bar for app-like products

---

### Trust Elements

#### Testimonials

- Use real, specific testimonials with: full name, title, company, and photo
- The best testimonial describes a specific result: "We reduced churn by 22% in 3 months"
- Do not use generic praise: "Great product!" is worth nothing; "This replaced our entire analytics stack" is gold
- For B2B: company size and industry context increases relevance
- For B2C: demographic similarity to target audience increases identification

#### Reviews

- Aggregate review scores (G2, Trustpilot, Capterra) are credible; show score + number of reviews
- Link to the review platform for third-party verification
- Display recency; "2,000 reviews" is stronger when paired with "4.8/5 as of 2024"

#### Client Logos

- Show logos of recognizable brands (the halo effect is real)
- Gray/monochrome logos look more professional and integrate better visually
- Maximum 6–8 logos on homepage; more dilutes impact
- Animated logo carousels for mobile only — desktop logos should be static and scannable

#### Certifications and Awards

- Place near relevant conversion actions (security badges near forms, compliance certs in B2B)
- Never cluster awards in a "wall of badges" — it looks desperate; curate the most relevant

#### Statistics

- Specific numbers carry more weight than rounded ones: "41,279 users" > "40,000+ users"
- Lead with impact metrics users care about: "Saves teams 8 hours per week on average"
- Source statistics where possible; unsourced claims read as marketing, not evidence

---

### Content Flow

The optimal homepage reading order follows user psychology — from awareness to intent:

1. **Hero** — What is this? Why should I care? What do I do next?
2. **Social proof bar** — Who else uses this? Am I in good company?
3. **Features / Capabilities** — What can it actually do?
4. **Benefits** — What does that mean for me specifically?
5. **How it works / Process** — Is this hard? Will I be able to implement this?
6. **Case studies or deeper proof** — Has this worked for someone like me?
7. **FAQ** — What are my remaining objections?
8. **Final CTA** — Decision point; the user has now been fully informed

> *Each section should answer the next logical question in the user's mind. When a section fails to do this, users scroll past or leave.*

---

### UX Considerations

#### Scanning Behavior

- Users scan, they do not read — design for scanning first, reading second
- F-pattern applies to text-heavy pages; Z-pattern applies to visual/marketing pages
- The first word of every paragraph carries disproportionate visual weight
- Use bold lead sentences for paragraph scanning

#### Visual Hierarchy

- Establish hierarchy through typographic contrast before applying color
- Ensure hierarchy is legible in grayscale — do not depend on color alone

#### White Space

- White space is not empty space; it is active space that creates breathing room and focus
- Increase line height (1.5–1.7 for body text), paragraph spacing, and section padding beyond what feels comfortable in the wireframe — designs always feel tighter when implemented
- Generous white space signals quality and confidence; cramped layouts signal low trust

#### Readability

- Body text minimum: 16px (browser default); 18px preferred
- Line length: 55–75 characters; never exceed 85 characters
- Sufficient contrast: minimum 4.5:1 for body text (WCAG AA); 7:1 for enhanced (WCAG AAA)
- Avoid centered text for more than 2 lines — it is difficult to read at length

## About Page Design Guidelines

### Purpose

The About page is frequently among the top 5 most visited pages on any website. Its function is not to list company facts — it is to:

1. **Build trust** by demonstrating real human beings are behind the product
2. **Humanize the brand** through authentic storytelling
3. **Demonstrate expertise** through evidence, not assertion
4. **Create alignment** by attracting the right customers, partners, and employees

> *An About page that reads like a press release has failed. An About page that makes someone feel something has succeeded.*

---

### Company Story

#### Origin Story

- Begin with the problem the founders or company set out to solve — not founding dates or legal incorporation
- The best origin stories follow a structure: **problem experienced → insight discovered → solution built**
- Be specific about the moment of founding; vague "passion for innovation" stories are invisible
- Authenticity signals: include failures, pivots, and hard lessons alongside successes

#### Mission

- A mission statement should be a single, specific sentence answering: What do we do? For whom? To what end?
- Avoid mission statement clichés: "world-class," "synergy," "disruptive," "empowering"
- Test it: could this mission statement belong to any other company? If yes, rewrite it.

#### Vision

- Vision is directional and aspirational — where is the company going in 5–10 years?
- Vision should feel ambitious enough to inspire but specific enough to be believed
- One to two sentences maximum; vision bloated into paragraphs loses meaning

#### Brand Values

- Limit to 4–6 values; more dilutes meaningfulness
- Each value should be: specific, actionable, and differentiated from competitors
- Illustrate each value with a concrete example, behavior, or story — not just a label and definition
- "We value transparency" is meaningless without an example of transparency in action

---

### Team Section

#### Team Photos

- Consistent photography style (same background, lighting, framing) communicates professionalism
- Natural, candid shots humanize better than stiff headshots — especially for culture-forward brands
- Photos must be high resolution; blurry or pixelated team photos destroy credibility
- Include photos of real people; stock photos of "team members" are immediately recognizable and damaging

#### Leadership Profiles

- Include: name, title, brief expertise summary (2–3 sentences), professional background
- Personal detail (interest, hobby, fun fact) increases memorability and approachability
- Link to LinkedIn for B2B contexts; it validates identity and expertise
- For founders: include why they founded the company; this is a trust signal

#### Expertise Presentation

- Surface domain expertise through credentials, publications, or speaking engagements
- Past employers or notable projects provide instant credibility context
- Certifications relevant to the business should appear in individual bios

---

### Credibility Elements

#### Achievements

- Be specific: "Named #1 in Customer Satisfaction by G2 for 6 consecutive quarters" not "Award-winning service"
- Recency matters: dated awards (5+ years old) can be counterproductive
- Third-party recognition carries more weight than self-reported achievement

#### Certifications and Compliance

- Display certifications relevant to your industry and audience (ISO, SOC 2, GDPR, etc.)
- Link to certification documentation where possible; claims without evidence are weak
- For B2B SaaS: security certifications near the trust section reduce enterprise sales friction

#### Partnerships

- Logos of technology or integration partners provide credibility by association
- Named strategic partnerships should include a brief description of the relationship

---

### Company Culture

#### Work Philosophy

- Articulate the working style: remote-first, async, flat hierarchy, etc.
- Prospective employees read About pages — culture content is a talent acquisition tool
- Be honest about the culture, including its tradeoffs; false positive culture content alienates the right people

#### Community Involvement

- Philanthropic activities and community contributions humanize the brand
- Specific impact (hours volunteered, donations made, projects supported) is more credible than vague commitment

---

### Timeline Section

- Use visual timelines for companies with 5+ years of meaningful history
- Include: founding, major product launches, funding milestones, acquisitions, team growth
- Timelines should emphasize transformation and progress, not just events
- Keep timelines concise; milestone fatigue sets in quickly

---

### About Page UX Best Practices

- **Authenticity**: Real photography > stock imagery everywhere on this page; invest in proper team photography
- **Storytelling**: Structure the page as a narrative with a beginning, middle, and implied future
- **Emotional connection**: What emotion should the user feel after reading this page? Design toward that emotion
- **Visual balance**: Alternate text-heavy and image-heavy sections to maintain visual rhythm
- **CTA**: Include a subtle next step — contact us, view our work, explore careers — at the bottom of the page

---

## Services Page Design Guidelines

### Purpose

The services page must accomplish three simultaneous objectives:

1. **Clarify what you offer** — users must understand the full scope of services without confusion
2. **Drive conversion** — guide qualified leads toward inquiry or purchase
3. **Reduce cognitive friction** — prevent decision paralysis from overwhelming choices or unclear differentiation

> *The services page is where interest becomes intent — and where poor UX most often kills the conversion.*

---

### Service Architecture

#### Categorization

- Group related services into 3–5 categories maximum; more creates analysis paralysis
- Category names should be benefit-oriented, not internally-coined jargon
- If you offer more than 8–10 distinct services, a visual filter or navigation component is required
- Test category names with users; internal naming conventions rarely match user mental models

#### Information Hierarchy

- Services most relevant to the target audience or highest-margin should appear first
- Do not rely on alphabetical ordering; user relevance should dictate sequence
- Summary cards on the main services page should link to dedicated service detail pages for complex offerings

#### Navigation Structure

- Anchor navigation (sticky sidebar or in-page nav) is essential for long services pages
- Provide a summary overview near the top allowing users to skip to their area of interest
- "Compare services" functionality is valuable when offerings have overlapping audiences

---

### Individual Service Blocks

Each service must answer the following questions in sequence:

1. **Title**: Clear, specific, jargon-free. "Brand Identity Design" not "Creative Vision Solutions"
2. **Problem statement**: Name the pain the user is experiencing before describing the solution
3. **Solution description**: How do you specifically address that problem? What is unique about your approach?
4. **Benefits**: Outcomes the client will experience — use bullet points with outcome-first language
5. **Deliverables**: What will the client actually receive? Be exhaustive; uncertainty kills conversions
6. **Process**: Brief overview of how the engagement works; reduces anxiety about "what happens next"
7. **Pricing guidance**: Price transparency dramatically increases conversion and lead quality; even a "starting from" number filters unqualified leads and builds trust
8. **CTA**: Specific to the service — "Schedule a brand discovery call" not generic "Contact us"

---

### Visual Design

#### Icons and Illustrations

- Icons should be consistent in style (line weight, fill vs. outline, corner radius) across all services
- Custom illustrations signal higher quality; generic icon libraries (Flaticon) signal commodity
- Icons clarify categorization; they do not substitute for copy
- For complex services, process diagrams or flowcharts reduce explanation burden on copy

#### Service Imagery

- Real project photography or case study imagery is significantly more persuasive than abstract visuals
- Before/after comparisons are highly effective for design, development, or transformation services
- Video walkthroughs of service processes outperform static imagery for complex or high-investment services

#### Comparison Tables

- Use when users are likely choosing between service tiers (Basic/Pro/Enterprise)
- Highlight the recommended tier visually; guide the decision
- Keep comparison rows to the most important differentiators — 8–12 rows maximum
- Mobile optimization of comparison tables is frequently neglected; vertical stacking with clear category labels is the fallback

---

### Conversion Elements

#### Testimonials

- Place service-specific testimonials near the relevant service block
- Results-oriented quotes outperform sentiment-based praise
- Video testimonials convert significantly higher than text; invest in 1–2 for flagship services

#### Case Studies

- Link to detailed case studies from the services page for high-consideration purchases
- Case study teasers (image + result + industry + brief description) are more scannable than full text
- Quantify results wherever possible: "43% increase in lead volume in 60 days"

#### FAQs

- Address the most common objections and questions for each service or service category
- FAQ copy reveals that you understand your customer's hesitations — this itself is a trust signal
- Use structured FAQ schema for SEO benefit

#### Lead Capture and CTAs

- High-friction CTAs ("Schedule a call," "Request a proposal") suit high-consideration services
- Low-friction CTAs ("Download our service guide," "Get a free estimate") suit exploratory users
- Consider a dual CTA strategy: one for ready-to-buy, one for still-evaluating users

---

### Services Page UX Requirements

- **Easy comparison**: Users should be able to understand differences between services within 60 seconds
- **Clear expectations**: Every service must communicate timeline, process, and deliverables; vagueness creates anxiety
- **Reduced decision fatigue**: If offering more than 5 services, provide a diagnostic tool or "Which service is right for me?" quiz

---

## Contact Page Design Guidelines

### Purpose

The contact page is a conversion-critical page often neglected in design investment. It must:

1. **Encourage communication** by reducing friction to near zero
2. **Generate qualified leads** by gathering the right information efficiently
3. **Build trust** at the moment of highest intent — the user is ready to act

> *The moment a user reaches the contact page, they have made a decision. Your job is not to persuade them further — it is to not stop them.*

---

### Contact Form Design

#### Number of Fields

- The ideal contact form has **3–5 fields** for general inquiries
- Each additional field reduces form completion rate by approximately 10–15%
- Only ask for information genuinely needed to respond usefully
- Do not ask for company size, budget, or phone number unless the business process absolutely requires it at this stage

#### Labels and Microcopy

- Use floating labels or always-visible labels; placeholder-only text fails accessibility requirements
- Label text should be plain language: "Your message" not "Inquiry description"
- Optional fields should be clearly marked `(optional)`; do not mark required fields with asterisks only
- Add guidance text below fields when format matters: "e.g., project@yourcompany.com"

#### Validation

- Validate on blur (when user leaves a field), not on submit — early validation feels adversarial
- Inline validation for email and phone fields reduces back-and-forth
- Never clear a filled-out form on validation error; this is among the most frustrating UX anti-patterns

#### Error States

- Error messages must appear adjacent to the field that caused them, not in a banner at the top
- Error text: explain what is wrong and how to fix it — "Please enter a valid email address" not "Invalid email"
- Error states: red border + error icon + red error text; never rely on color alone (accessibility)
- Success state must be unambiguous — full page confirmation or prominent inline confirmation with specific follow-up expectation

#### Success States

- Confirm submission immediately and visibly
- Set expectations: "We'll respond within 1 business day" not "We'll be in touch"
- Optionally: redirect to a dedicated thank-you page (enables Google Analytics goal tracking)
- Send an automatic confirmation email when possible — it signals professionalism and provides reference

---

### Contact Methods

Provide multiple contact channels; different users prefer different communication modes:

| Method | When to Include | Notes |
|---|---|---|
| **Email** | Always | Display as a mailto link; do not hide behind contact forms only |
| **Phone** | When voice is viable | Include business hours and time zone |
| **Live Chat** | When support/sales team is available | Never show chat widget when offline without a clear offline message |
| **Social Media** | When actively monitored | Only list channels with <24hr response time; dead channels damage trust |
| **Physical Address** | For local businesses; trust for SaaS | Required for legal compliance in many jurisdictions |

---

### Trust Signals on the Contact Page

- **Response time**: "We respond to all inquiries within 4 business hours" — specificity builds trust
- **Business hours**: Clearly stated with time zone; international users need this
- **Team photo or personal headshot**: A face associated with the contact removes anonymous corporate distance
- **Privacy assurance**: A one-line statement near the form ("We never share your information") reduces hesitation
- **Security indicators**: For forms collecting sensitive data, HTTPS indicator and brief privacy statement are essential

---

### Contact Page UX Best Practices

- Remove the header and footer navigation from the contact page to eliminate distraction during high-intent actions (optional but tested to improve conversion)
- Never require account creation before allowing contact
- Auto-fill support: ensure fields work with browser autofill (correct `autocomplete` attributes)
- Honeypot fields for spam prevention over CAPTCHA; CAPTCHAs create significant friction and accessibility issues
- Mobile usability: test the form on real devices; keyboard behavior, tap target size, and autocomplete must all function correctly

---

### Map and Location

**Include a map when:**
- You have a physical location users may visit
- Location is a trust signal (e.g., verifying you are a real, local business)
- The business serves a specific geography

**Map UX requirements:**
- Use an embedded map but do not depend on it for address information — always display the address in plain text
- Provide parking, transit, or wayfinding instructions for physical destinations
- Map should never be the only method of conveying address; screen readers and users with slow connections need text alternatives

## Accessibility Standards

*Compliance target: WCAG 2.2 Level AA minimum; Level AAA where feasible for critical content.*

### Color Contrast

| Text Type | Minimum Contrast Ratio |
|---|---|
| Normal body text (< 18pt) | 4.5:1 (AA) / 7:1 (AAA) |
| Large text (≥ 18pt or 14pt bold) | 3:1 (AA) / 4.5:1 (AAA) |
| UI components and graphic elements | 3:1 |
| Decorative elements | No requirement |

- **Tools**: Use Stark (Figma), Colour Contrast Analyser, or browser DevTools accessibility panel
- Never use color as the only means of conveying information (e.g., error states must use icon + color + text)
- Ensure contrast holds in dark mode, on low-brightness screens, and under ambient lighting conditions

---

### Keyboard Navigation

- All interactive elements must be reachable and operable via keyboard (`Tab`, `Shift+Tab`, `Enter`, `Space`, `Escape`, `Arrow keys`)
- Tab order must be logical and follow visual reading order
- Modal dialogs must trap focus within the modal while open; return focus to the trigger on close
- Dropdown menus, accordions, and carousels must have defined keyboard interaction patterns (WAI-ARIA Authoring Practices)
- Skip-to-main-content link must be the first focusable element on every page

---

### Screen Reader Compatibility

- Use semantic HTML elements (`<nav>`, `<main>`, `<header>`, `<footer>`, `<article>`, `<section>`) as the first tool; ARIA only to supplement gaps
- All images must have descriptive `alt` text; decorative images use `alt=""`
- Form inputs must be programmatically associated with their labels via `for`/`id` pairing or `aria-labelledby`
- Icon-only buttons must have an accessible name via `aria-label`
- Dynamic content changes must be announced via `aria-live` regions
- Error messages must be associated with the relevant form field via `aria-describedby`

---

### Focus States

- The default browser focus outline is unacceptable in most design systems; design a custom focus indicator
- Focus indicator minimum: 2px offset outline in a high-contrast color (typically the brand primary or black/white inverse)
- Focus state must be visible across all interactive elements: links, buttons, inputs, selects, checkboxes, radio buttons, custom components
- WCAG 2.2 requires focus indicators to have sufficient area and contrast; consult criterion 2.4.11 and 2.4.12 for specifics

---

### Form Accessibility

- Every input must have a visible, associated label
- Required fields must be indicated in text (not asterisk alone); explain asterisk meaning if used
- Fieldsets and legends for grouped controls (radio buttons, checkboxes)
- `autocomplete` attributes on appropriate fields (name, email, phone, address)
- Error identification: clearly identify the field in error, describe the nature of the error, and suggest how to fix it

---

### Semantic Structure

- Heading levels must be hierarchically logical (H1 → H2 → H3); never skip levels for visual effect
- Use `<button>` for actions, `<a>` for navigation; never the reverse
- Lists must use `<ul>` or `<ol>` appropriately; do not simulate lists with line breaks
- Tables must use `<th>` with `scope` attributes; provide `<caption>` for context
- Landmark regions allow screen reader users to navigate by page structure; every page must have the major landmarks defined

## Responsive Design Standards

### Breakpoint Strategy

| Breakpoint | Range | Target Devices |
|---|---|---|
| **xs** | 320–479px | Small phones (SE, budget Android) |
| **sm** | 480–767px | Standard phones |
| **md** | 768–1023px | Tablets (portrait) |
| **lg** | 1024–1279px | Tablets (landscape), small laptops |
| **xl** | 1280–1535px | Standard desktop |
| **2xl** | 1536px+ | Large desktop, ultrawide |

**Recommended CSS custom properties:**
```css
--bp-sm: 480px;
--bp-md: 768px;
--bp-lg: 1024px;
--bp-xl: 1280px;
--bp-2xl: 1536px;
```

---

### Mobile (320–767px)

- Single-column layout; no multi-column grids above 2 columns
- Navigation collapses to hamburger menu or bottom navigation
- Touch targets minimum 44×44px with sufficient spacing between targets
- Font sizes: heading 28–36px, body 16–18px
- Images: full-width or constrained with `max-width: 100%`
- Remove or collapse secondary content; focus on primary user tasks
- Horizontal scroll is a failure state — never allow it except in explicitly scrollable containers
- Form inputs: `font-size: 16px` minimum to prevent iOS auto-zoom on focus

---

### Tablet (768–1023px)

- 2-column grids common; 3-column acceptable for card grids
- Navigation may remain visible or collapse; context-dependent
- Touch-first still applies; hover is unreliable
- Sidebar patterns work at this size but require careful consideration of content priority
- Font sizes: heading 32–48px, body 16–18px

---

### Desktop (1024–1535px)

- Full navigation visible
- 3–4 column grids for card components
- Content max-width container: 1200–1440px centered with auto margins
- Sidebar + main content layouts fully supported
- Hover states and interactive micro-interactions fully expressed

---

### Large Screens (1536px+)

- Never let content stretch edge-to-edge; use max-width containers
- Increase line height and font size slightly for very large displays
- Background decorative elements can become more elaborate at large sizes
- Two-page spreads for product showcases and case studies become viable

---

### Responsive Typography

Use `clamp()` for fluid type that scales gracefully between breakpoints:

```css
/* Fluid heading — scales from 28px at 320px viewport to 56px at 1280px */
font-size: clamp(1.75rem, 4vw + 0.5rem, 3.5rem);

/* Body text — stable at 16–18px */
font-size: clamp(1rem, 1.5vw + 0.75rem, 1.125rem);
```

---

### Layout Recommendations

- CSS Grid for page-level layout; Flexbox for component-level layout
- Avoid fixed heights on containers that hold variable content
- Define component minimum and maximum sizes, not just target sizes
- Always test with real content at all breakpoints; placeholder text does not reveal wrapping failures

---

## Design System Standards

### Typography Scale

Establish a modular type scale. The **Major Third (1.250)** ratio is recommended for most products; **Minor Third (1.200)** for dense information products.

| Token | Size (Major Third) | Usage |
|---|---|---|
| `--text-xs` | 10px | Legal, captions, overlines |
| `--text-sm` | 12px | Labels, metadata, badges |
| `--text-base` | 16px | Body text (baseline) |
| `--text-lg` | 20px | Large body, lead paragraphs |
| `--text-xl` | 24px | H4, card headings |
| `--text-2xl` | 30px | H3 |
| `--text-3xl` | 38px | H2 |
| `--text-4xl` | 48px | H1 |
| `--text-5xl` | 60px | Hero headlines |
| `--text-6xl` | 76px | Display text |

**Font selection rules:**
- Maximum 2 typefaces: one display/heading, one body
- Body font: high x-height, generous spacing, designed for screen reading
- Display font: distinctive enough to communicate brand; legible at all heading sizes
- Always define a system font fallback stack

---

### Color System

#### Token Architecture

```css
/* Primitives (raw values) */
--blue-500: #2563EB;
--gray-900: #111827;

/* Semantic tokens (context-derived) */
--color-primary: var(--blue-500);
--color-background: var(--gray-50);
--color-text-primary: var(--gray-900);
--color-text-secondary: var(--gray-600);
--color-border: var(--gray-200);
--color-error: var(--red-600);
--color-success: var(--green-600);
--color-warning: var(--amber-500);
```

#### Color Roles

| Role | Purpose |
|---|---|
| **Primary** | Key actions, links, active states |
| **Secondary** | Supporting actions, hover states |
| **Accent** | Highlights, badges, notifications |
| **Neutral** | Text, borders, backgrounds |
| **Success** | Confirmation, positive states |
| **Warning** | Caution, degraded states |
| **Error** | Failures, destructive states |
| **Info** | Neutral informational states |

**Rule**: Never add a color to the system to solve a one-off design problem. Solve the design problem first.

---

### Spacing System

Use a base-4 or base-8 spatial system. Base-8 recommended for most products:

```css
--space-1:  4px;
--space-2:  8px;
--space-3:  12px;
--space-4:  16px;
--space-5:  24px;
--space-6:  32px;
--space-7:  48px;
--space-8:  64px;
--space-9:  96px;
--space-10: 128px;
```

**Rules:**
- Use only system spacing values; do not use arbitrary pixel values
- Internal component spacing: `space-1` through `space-4`
- Component-to-component spacing: `space-4` through `space-6`
- Section-level spacing: `space-8` through `space-10`

---

### Grid System

```css
/* 12-column grid — industry standard */
--grid-columns: 12;
--grid-gutter: var(--space-5);   /* 24px */
--grid-margin: var(--space-6);   /* 32px on tablet, 64px on desktop */
--container-max: 1280px;
```

| Breakpoint | Columns | Gutter | Margin |
|---|---|---|---|
| Mobile | 4 | 16px | 16px |
| Tablet | 8 | 24px | 32px |
| Desktop | 12 | 24px | 64px |

### Component Standards

#### Buttons

Every button must have: default, hover, focus, active, disabled, and loading states defined.

| Variant | Use Case |
|---|---|
| **Primary** | Single primary action per page/section |
| **Secondary** | Supporting or alternative actions |
| **Ghost** | Low-emphasis; tertiary actions |
| **Danger** | Destructive, irreversible actions |
| **Link** | Inline navigation; minimal visual weight |

**Sizing:**
- Small: 32px height, `text-sm`, `space-3` horizontal padding
- Medium (default): 40px height, `text-base`, `space-4` horizontal padding
- Large: 48px height, `text-lg`, `space-5` horizontal padding

**Rules:**
- Full-width buttons on mobile only; auto-width on desktop
- Loading state must indicate something is happening (spinner + label change: "Submitting…")
- Disabled state: 50% opacity, no hover effect, `cursor: not-allowed`

---

#### Form Standards

| Element | Standard |
|---|---|
| **Label** | Always visible; never placeholder-only |
| **Input height** | 40px (medium), 48px (large/prominent) |
| **Border** | 1px solid neutral border color |
| **Focus border** | 2px solid primary color |
| **Error border** | 2px solid error color + error message below |
| **Corner radius** | Match overall system (4–8px typical) |
| **Spacing between fields** | `space-5` (24px) minimum |

All form elements must have visible focus states and be keyboard-navigable.

---

#### Card Standards

Cards are containers for related content; they must be designed consistently across the product.

**Anatomy:**
- Optional image or media area (top, 16:9 or 4:3 ratio)
- Padding: `space-5` (24px) internal
- Border radius: match system token
- Shadow: define `shadow-sm`, `shadow-md`, `shadow-lg` tokens for card elevation
- Interactive cards (linking) must have hover and focus states on the entire card surface
- Never use multiple interactive elements inside a card that links as a whole — it creates conflicting click targets

---

### Documentation Standards

Every component in the system must be documented with:

1. **Component name** and description
2. **When to use / when not to use**
3. **Props/variants** table
4. **All interactive states** (visual + description)
5. **Accessibility notes** (ARIA roles, keyboard behavior)
6. **Do/Don't examples**
7. **Related components**

> *Undocumented components do not exist in a design system. If the documentation isn't there, designers and developers will reinvent it differently every time.*

## Quick Reference Checklists

### Homepage Launch Checklist

- [ ] Value proposition clear within 5 seconds without scrolling
- [ ] Primary CTA visible above the fold on all devices
- [ ] Social proof present in the hero section
- [ ] Mobile navigation tested on real devices
- [ ] Page load time under 2.5s (LCP)
- [ ] Hero image optimized (WebP, correct dimensions, lazy loading for below-fold images)
- [ ] All links functional; no broken paths
- [ ] Forms tested end-to-end including confirmation state
- [ ] Analytics and conversion tracking verified

### Accessibility Audit Checklist

- [ ] Contrast ratios meet WCAG AA for all text
- [ ] All images have appropriate `alt` text
- [ ] Keyboard navigation tested through all interactive paths
- [ ] No keyboard traps (except intentional modal traps)
- [ ] Focus indicators visible on all interactive elements
- [ ] Screen reader tested with VoiceOver (macOS/iOS) or NVDA (Windows)
- [ ] Form fields have associated labels
- [ ] Error messages are descriptive and programmatically associated

### Responsive Design Checklist

- [ ] Tested at 320px, 375px, 768px, 1024px, 1280px, 1536px
- [ ] No horizontal scroll at any breakpoint
- [ ] Touch targets ≥ 44×44px on mobile
- [ ] Font sizes ≥ 16px for body on mobile
- [ ] Navigation usable at all breakpoints
- [ ] Images and media responsive and not distorted
- [ ] Tables have a mobile-optimized fallback
